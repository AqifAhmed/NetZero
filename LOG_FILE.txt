STEP 1 - Project Setup (Expo + React Native)

STEP 2 - Folder Structure & Architecture Planning

STEP 3 - Data Model & Category Definitions
  - Designed a flat expense object structure:
      { id, title, amount, category, date, createdAt }
  - id is a timestamp string (Date.now().toString()) — simple unique key
  - amount is stored as a number in PKR
  - date is stored as a 'yyyy-MM-dd' string (date-fns format)
  - category is one of: 'Food', 'Transport', 'Shopping', 'Bills', 'Other'
  - Each category has a fixed color and icon defined in theme.js:
      Food → #EF4444 / restaurant
      Transport → #38BDF8 / car
      Shopping → #FBBF24 / bag-handle
      Bills → #86EFAC / receipt
      Other → #94A3B8 / ellipsis-horizontal-circle

STEP 4 - Global State with Context API + useReducer
  - Created ExpensesContext.js using createContext()
  - State is managed through useReducer instead of multiple useState calls
    because the state has multiple related fields that change together
  - Defined five reducer actions:
      LOAD_ALL   — restores expenses + selectedMonth/Year from AsyncStorage
      ADD        — appends a new expense object to the array
      UPDATE     — replaces an existing expense matched by id
      DELETE     — filters out the expense with the given id
      SET_MONTH  — updates selectedMonth and selectedYear for navigation
  - Context exposes derived values (not just raw state):
      filteredExpenses  — expenses filtered to the selected month/year
      totalThisMonth    — sum of amounts for those expenses
      spendByCategory   — object with per-category totals for the month
  - Also exposes selectedMonth, selectedYear, loading, and action functions:
      addExpense, updateExpense, deleteExpense, setMonth

STEP 5 - AsyncStorage Persistence
  - On mount, a useEffect in ExpensesContext dispatches LOAD_ALL
    after reading two keys from AsyncStorage:
      'expenses'     — JSON array of all expense objects
      'selectedDate' — JSON object with month and year
  - A separate useEffect watches the expenses array; whenever it changes,
    it serializes and saves the updated array back to AsyncStorage
  - This means add/update/delete operations automatically persist without
    any extra calls — the save is triggered reactively by state change
  - loading state starts as true and is set to false after LOAD_ALL runs;
    screens show a loading indicator until this is done

STEP 6 - Bottom Tab Navigation
  - Created MainTab.js using createBottomTabNavigator from React Navigation
  - Three tabs: Home (home-outline), Expenses (list-outline), Summary (pie-chart-outline)
  - Tab bar styled with dark navy background and green active tint
  - App.js wraps MainTab in NavigationContainer and provides ExpensesProvider
    so context is available to all screens and components
  - Safe area handling is done via SafeAreaProvider from react-native-safe-area-context

STEP 7 - HomeScreen

STEP 8 - ExpensesScreen

STEP 9 - SummaryScreen

STEP 10 - ExpenseCard Component

STEP 11 - ExpenseModal Component (Add + Edit)

  - Fields:

  - Form validation: title and amount must not be empty before submitting

PROBLEMS FACED


Problem 1: useReducer initial state not loading from AsyncStorage
  - On first render, expenses showed as empty even after AsyncStorage had data
  - The issue was that the LOAD_ALL dispatch was async but the initial render
    happened before the AsyncStorage.getItem() Promise resolved
  - This caused a flash of empty content every time the app opened

Problem 2: Pie chart crashing when all expenses are zero
  - react-native-chart-kit's PieChart throws an error if the data array
    is empty or all values are zero
  - When a month had no expenses the Summary screen would crash on open

Problem 3: Month rollover going negative
  - Pressing the left arrow on January decremented month to 0
  - JavaScript's Date and date-fns both use 1-based months, so month 0
    caused invalid date formatting and wrong filter results

Problem 4: Modal form not resetting after submission
  - After adding an expense and closing the modal, reopening it showed
    the previously entered values still in the fields
  - useState for form fields was not being reset after the modal closed

Problem 5: Category filter chip staying selected after changing month
  - If "Food" was selected as the active filter on ExpensesScreen and the
    user changed the month from HomeScreen, the filter chip remained active
  - The filtered list showed 0 results even when the new month had Food expenses
    because the category filter was not resetting on month change

Problem 6: Amount field accepting non-numeric input
  - Using keyboardType='numeric' on the amount TextInput still allowed
    characters like '.' to be entered multiple times (e.g. "45..0")
  - Submitting with a malformed amount stored NaN in AsyncStorage


SOLUTIONS

Solution 1 (AsyncStorage loading):
  - Added a loading boolean to the reducer state (initially true)
  - LOAD_ALL sets loading to false after populating the state
  - All three screens check the loading flag and render an ActivityIndicator
    while it is true, preventing any flash of empty content

Solution 2 (Pie chart crash):
  - Added a guard before rendering the chart:
    if spendByCategory has no entries with amount > 0, render a
    "No expenses this month" placeholder Text instead of the PieChart

Solution 3 (Month rollover):
  - Rewrote the setMonth logic with explicit bounds:
      if (newMonth < 1)  → newMonth = 12, newYear = year - 1
      if (newMonth > 12) → newMonth = 1,  newYear = year + 1
  - Tested by pressing the left arrow 13 times from any starting month

Solution 4 (Form reset):
  - Added a useEffect inside ExpenseModal that watches the 'visible' prop
  - When visible changes to false, all form state (title, amount, date,
    category) is reset to default values
  - This ensures the modal is always clean when opened again

Solution 5 (Category filter reset):
  - Moved the activeCategory state up one level so it resets to 'All'
    whenever selectedMonth or selectedYear changes in the context
  - Used a useEffect in ExpensesScreen that depends on [selectedMonth,
    selectedYear] and calls setActiveCategory('All')

Solution 6 (Amount validation):
  - Added a parseFloat check in the onSubmit handler:
      if (isNaN(parseFloat(amount)) || parseFloat(amount) <= 0) → show error
  - Also stripped any non-numeric characters except one decimal point
    using a regex replace inside the onChange handler before setting state


LEARNING REFLECTION

What I understood:
  - How useReducer is better than multiple useState calls when state values
    are related — grouping them in one reducer made LOAD_ALL, ADD, UPDATE,
    and DELETE much cleaner than updating 4 separate state variables
  - How React Native's component model differs from React web — no div/span,
    everything is View/Text/TouchableOpacity, and StyleSheet.create() replaces CSS
  - How AsyncStorage works as a key-value store for local persistence —
    serializing to JSON on save and parsing on load is the standard pattern
  - How derived state (filteredExpenses, totalThisMonth, spendByCategory)
    should be computed inside the context rather than in each screen,
    so screens stay simple and just consume ready-to-use values
  - How bottom tab navigation is configured with createBottomTabNavigator
    and how to customize tab bar appearance (colors, icons, label style)
  - How to build a custom calendar grid in React Native using a 2D array
    of week rows and computed day-of-week offsets
  - How react-native-chart-kit PieChart needs non-zero data and how to
    guard against crashes with empty datasets

What was difficult:
  - The biggest challenge was the async gap between component mount and
    AsyncStorage loading. It took a while to understand why loading: true
    in the initial reducer state was the right fix rather than trying to
    await inside the component body.
  - Building the custom calendar picker was more involved than expected —
    computing which column the 1st of each month falls on, filling the
    trailing days, and disabling future cells all had edge cases.
  - Getting the pie chart slices to use the exact same colors as the
    category chips required carefully matching the data array order to
    the CATEGORIES constant. Any mismatch caused wrong colors on slices.
  - Managing the interaction between the month selector (on Home) and
    the category filter chip (on Expenses) across two separate screens
    required thinking carefully about where state should live.

Overall:
  This project brought together everything from the semester — component
  architecture, props, state, context, navigation, and async data — in one
  real app. The biggest shift from Assignment 2 was moving from useState
  to useReducer for complex state, and from an API call to local AsyncStorage
  for persistence. Building something that actually runs on a phone and saves
  data across restarts felt like a significant step up from the web assignments.
